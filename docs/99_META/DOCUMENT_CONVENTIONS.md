# Document Conventions

> *"Consistency is one of the highest forms of engineering quality."*

---

# Purpose

This document defines the writing, formatting, metadata, and structural conventions used throughout the AI Architect Handbook.

Its objective is to ensure that every chapter feels like it belongs to a single engineering organization.

---

# Standard Chapter Structure

Every chapter should follow this order:

1. Title
2. Purpose
3. Why This Exists
4. Philosophy
5. Principles
6. Architecture or Framework
7. Implementation (if applicable)
8. Trade-offs
9. Related Chapters
10. Living Chapter

---

# Metadata Standard

Each document should begin with metadata similar to:

- Title
- Version
- Status
- Owner
- Last Updated
- Depends On
- Influences

---

# Writing Style

Preferred:

- Clear
- Educational
- Evidence-based
- Timeless
- Professional
- Precise

Avoid:

- Marketing language
- Buzzwords
- Self-promotion
- Technology hype
- Unnecessary adjectives

---

# Naming Conventions

Use descriptive names.

Prefer:

- `PRODUCT_STRATEGY.md`
- `AI_ARCHITECTURE.md`

Avoid:

- `notes.md`
- `final2.md`
- `misc.md`

Use numeric prefixes when reading order matters.

---

# Diagrams

Use simple text diagrams whenever possible.

Diagrams should explain architecture rather than decorate documentation.

---

# Cross References

Avoid duplicating concepts.

Reference the canonical chapter instead.

The handbook follows the Single Source of Truth principle.

---

# Versioning

```text
Draft -> Review -> Approved -> Archived
```

Every significant update should include a brief change summary.

---

# Documentation Quality Checklist

Before publishing, verify:

- Clear purpose
- Logical structure
- Accurate terminology
- Consistent formatting
- Cross references
- No duplication
- Evidence where appropriate

---

## Living Document

Engineering evolves continuously.

This document captures current principles rather than immutable truths.

When new evidence contradicts an existing belief, the document should be updated accordingly.

Strong engineering values learning over consistency.
